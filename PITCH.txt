ONE-LINER

0GMind turns any AI agent's memory into a portable, ownable
asset. Store a context once on 0G, get one blob ID, and any

----------------------------------------------------------------
THE 30-SECOND PITCH
----------------------------------------------------------------

Every AI agent today is born with amnesia.

Switch from GPT to Claude? Memory gone.
Restart your agent? Memory gone.
Want a second agent to continue the work? You copy-paste a
giant prompt and pray.

AI memory is locked inside one provider, one session, one
silo. It is not portable, not owned, and not verifiable.

0GMind fixes this. We make memory a first-class, on-chain
asset:

  - STORED on 0G Storage (encrypted, decentralized, permanent)
  - PROCESSED on 0G Compute (sealed inference, glm-5)
  - OWNED on 0G Chain (every context is an ERC-721 NFT)
  - AUDITED on 0G DA (every access logged on-chain, forever)

One blob ID is all you hand off. Any agent loads it and
instantly knows everything the previous agent knew.


----------------------------------------------------------------
THE PAIN POINT (why this matters)
----------------------------------------------------------------

1. MEMORY DIES WITH THE MODEL
   Context windows are temporary. Close the tab, lose the
   brain. There is no "save file" for an AI's mind.

2. MEMORY IS NOT PORTABLE
   You cannot take a Claude conversation and continue it in
   GPT, or hand your agent's state to a teammate's agent.

3. MEMORY IS NOT OWNED
   The context you spent hours building belongs to the
   provider, not to you. You cannot sell it, license it,
   transfer it, or revoke it.

4. NO AUDIT TRAIL
   You have zero proof of who accessed your agent's memory
   or when.

These are not small UX problems. As the world moves to
multi-agent systems, portable + ownable + verifiable memory
becomes critical infrastructure. That layer does not exist
today. 0GMind is that layer.


----------------------------------------------------------------
HOW 0GMIND SOLVES IT (the magic)
----------------------------------------------------------------

We treat an AI context like a file with an address.

  context  ->  encrypt  ->  0G Storage  ->  blob ID
                                              |
                                              v
                                       mint NFT on 0G Chain
                                              |
                                              v
              any agent  <-  load by blob ID  <-  0G Compute

The blob ID is the portable memory address. Whoever holds it
(and has access rights) can rehydrate the full agent state
into ANY model through 0G Compute — no provider lock-in.


----------------------------------------------------------------
WHY WE USE EVERY LAYER OF 0G (not a bolt-on)
----------------------------------------------------------------

This is the key point for judging: remove any single 0G layer
and the product breaks. We are not "using 0G for storage" —
we are using the FULL stack as designed.

  0G STORAGE  — the memory itself. Encrypted context blobs
                live here. Decentralized and permanent. This
                is what makes memory portable.

  0G COMPUTE  — the brain. Inference runs on 0G's
                decentralized GPU network (glm-5). For private
                contexts, inference happens inside a TEE so
                even node operators never see the data.

  0G CHAIN    — the ownership. Each context is an ERC-721 NFT.
                Transfer it, license it, grant access, revoke
                access — all on-chain.

  0G DA       — the trust. Every access fires an on-chain
                ContextAccessed event. A permanent,
                uncensorable audit trail of who used your
                memory.

HOW WE LOAD THE DATA (the technical truth, said simply)
----------------------------------------------------------------

- When you STORE: the context is (optionally encrypted),
  written to 0G Storage, and the storage root hash becomes the
  blob ID. That blob ID is then registered on 0G Chain as an
  NFT you own.

- When you LOAD: we take the blob ID, fetch the blob back from
  0G Storage, decrypt it if private, and feed it as the system
  context to glm-5 on 0G Compute. The model answers as if it
  had the full prior memory.

- Right now we store the context as PLAIN TEXT (conversations,
  system prompts, agent state). The exact same pipeline works
  for FILES — PDFs, JSON memory dumps, embeddings, model
  checkpoints — because 0G Storage is file-native. Text is
  simply our v1 demo surface; file upload is a drop-in next
  step with zero architecture change.
